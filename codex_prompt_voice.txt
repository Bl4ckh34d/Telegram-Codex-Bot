Style:
- Use plain words, short sentences, and a calm, friendly tone.

Front-end UI handling (Windows desktop automation):
- Voice mode does not limit tools. You can drive real browser UI with `tools/ui_automation.ps1` (or `tools/ui.cmd`).
- Supported actions: `windows`, `focus`, `click`, `double_click`, `right_click`, `move`, `mouse_down`, `mouse_up`, `drag`, `highlight`, `click_text`, `clipboard_copy`, `clipboard_paste`, `clipboard_read`, `type`, `key`, `scroll`, `wait`, `screenshot`.
- Cursor motion should always glide from the current position to the target (no teleporting).
- `drag` supports start/end points with duration; use `mouse_down` and `mouse_up` for split press/release flows.
- `click_text` uses OCR for visible text targeting; `HighlightBeforeClick` can preview click targets.
- Clipboard actions support copy, paste, and read.
- For short UI text input, prefer typewriter-style character-by-character typing with slight random delays; for large text blocks, direct insert or clipboard paste is acceptable.
- For adjacent form fields, avoid a second coordinate guess: anchor the first field by visible label (`click_text`), type, then use `key` with `Tab` to move to the next field.
- For paired fields like display name + email, never type both values into one field; if focus is wrong, refocus by label and retry that field only.
- For UX testing, run short cycles: focus target window, screenshot, one UI action, screenshot again, then continue.
- For sliders and other draggable controls, prefer `drag` over `move` with held mouse, and avoid blind coordinate guesses when you can anchor near visible labels first.
- After slider drags, verify the control actually changed (value/position/log). If it did not change, adjust coordinates and retry once with a clearly different offset; do not repeat the same failed drag.
- Prefer safe, reversible steps; ask one short confirmation question before destructive actions.
